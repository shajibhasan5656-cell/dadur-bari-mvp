# Dadur Bari Implementation Rules

## 1. Core Rule: Follow the SRS Exactly

All implementation work must follow the Dadur Bari SRS without deviation. The final system must reflect the documented business model, brand identity, user experience, admin requirements, security rules, and operational goals.

## 2. Do Not Simplify the Product

The platform must not be simplified to a basic storefront. The complete commerce system described in the SRS must be preserved, including:

- Product and inventory complexity
- Admin operations
- Customer account features
- Checkout and payment handling
- SEO and analytics expectations
- Security and performance requirements

## 3. Do Not Skip Features

No required module may be removed, deferred, or reduced without explicit approval. This includes:

- Admin panel modules
- Customer dashboard modules
- Product variant handling
- Order workflow and payment workflow
- SEO and analytics features
- Security and audit requirements

## 4. Do Not Redesign the Brand Identity

The brand identity must remain consistent with the approved Dadur Bari direction:

- Premium
- Elegant
- Minimal
- Modern
- Trustworthy
- Comfortable
- Confident
- Timeless

The color palette, typography, and tone must remain aligned with the SRS.

## 5. Do Not Add Extra Features Without Approval

Any additional feature beyond the SRS must not be introduced unless approved by the project owner or product lead. This prevents scope drift and protects the planned implementation flow.

## 6. Ask Before Changing Business Logic

Business rules such as:

- Pre Order priority
- Product categories
- Delivery rules
- Payment handling
- COD conditions
- Inventory behavior
- Discount and coupon behavior

must not be changed casually. Any change to business logic must be discussed before implementation.

## 7. Preserve the Current Working MVP Until Final Replacement Is Approved

The existing MVP pages and current app behavior must remain untouched during planning and documentation work. The final implementation must only replace them after approval and after the staged build is ready.

## 8. Work in Stages

Implementation must proceed in stages rather than as a single large release. Each stage should complete and validate before moving to the next.

## 9. Explain What Will Change Before Code Changes

Before any implementation step, the planned change must be clearly explained. This includes:

- Which module is changing
- Why it is changing
- What user or admin impact it has
- Whether the current MVP is affected

## 10. Every Stage Must Build Successfully

Each implementation stage must be verified before moving forward. The project must remain in a shippable state after every stage.

## 11. Prefer Configurable Business Rules

The system must avoid hardcoded business logic in the final version. Business decisions should be editable from the admin panel wherever possible.

## 12. Maintain Security and Performance Standards

All implementation work must preserve the required security and performance expectations. Security must not be weakened for convenience, and performance must not degrade below the stated targets.

## 13. Maintain Quality and Accessibility

Implementation must preserve strong UI quality, mobile responsiveness, accessibility standards, and SEO readiness.

## 14. Documentation Must Be Kept in Sync

Any major implementation decision must be reflected in the planning documents so that the architecture and roadmap remain consistent.
